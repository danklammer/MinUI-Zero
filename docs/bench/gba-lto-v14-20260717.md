# GBA core LTO A/B (Brick, 2026-07-17)

## Question

Does link-time optimization reduce GBA core work enough to justify changing the v1.4 core
builds, without harming delivery or raising the clock selected by the governor?

## Method

- Device: TrimUI Brick, stock clocks, battery 100% to 97%, `Discharging` throughout.
- Frontend: release/v1.4 `38208374`.
- Game: F-Zero - Maximum Velocity (USA); original SRAM restored before every arm.
- Four 90-second arms per core in ABBA order: baseline, LTO, LTO, baseline.
- Baseline and candidate differed only by `-flto` on compilation and linking.
- Analysis includes only complete 60-frame windows and discards each arm's first ten windows
  plus the partial scripted-quit window. This leaves 9,480-9,600 measured frames per variant.
- The PMIC's `current_now` node is empty, so this test can establish CPU work, selected clock,
  delivery health, temperature, and binary size, but not short-window energy.

## Results

| Core | Metric | Baseline | LTO | Delta |
|---|---:|---:|---:|---:|
| gpSP | binary bytes | 938,976 | 871,600 | **-7.2%** |
| gpSP | median frame work | 3,640 us | 3,514 us | **-3.5%** |
| gpSP | median p95 work | 4,760 us | 4,294 us | **-9.8%** |
| gpSP | worst p99 window | 9,771 us | 9,174 us | **-6.1%** |
| gpSP | average sampled clock | 441,811 kHz | 441,600 kHz | -0.05% |
| gpSP | over-budget frames | 0 / 9,540 | 0 / 9,600 | no regression |
| gpSP | in-run audio underruns | 0 | 0 | no regression |
| gpSP | mean / peak temperature | 33.03 / 34 C | 33.05 / 34 C | noise |
| mGBA | binary bytes | 3,320,680 | 2,593,224 | -21.9% |
| mGBA | median frame work | 10,804 us | 10,771 us | -0.3% |
| mGBA | median p95 work | 12,790 us | 13,108 us | **+2.5%** |
| mGBA | average sampled clock | 744,152 kHz | 800,506 kHz | **+7.6%** |
| mGBA | over-budget frames | 46 / 9,480 | 61 / 9,480 | **0.485% -> 0.643%** |
| mGBA | in-run audio underruns | 0 | 5 | **regression** |
| mGBA | mean / peak temperature | 34.26 / 36 C | 34.43 / 36 C | noise |

## Verdict

- **Ship gpSP LTO.** Both repetitions agree: lower typical and tail CPU work, unchanged clock,
  zero delivery regressions, a smaller binary, and a clean 20-second functional smoke test.
- **Reject mGBA LTO.** The size reduction does not translate into less CPU work and coincides
  with a higher selected clock, more over-budget frames, and five in-run audio underruns.
- Do not claim a battery-life percentage from this run. The available capacity counter is too
  coarse and instantaneous current is unavailable.
